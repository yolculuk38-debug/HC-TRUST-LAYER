# HC-TRUST-LAYER Trust Graph Foundation

## Purpose

This document defines a foundational trust graph model for HC-TRUST-LAYER.

It describes conceptual relationships for provenance, witness, validator, and canonical record linkage inside verification infrastructure.

This is a protocol-level design reference only:
- no production-ready claim
- no decentralization claim
- no blockchain claim
- no cryptographic implementation in this document

## Trust Graph Overview

The HC-TRUST-LAYER trust graph is a directed relationship model connecting:
- canonical record nodes
- provenance event nodes
- witness attestation nodes
- validator assessment nodes
- audit trail linkage nodes

The trust graph supports transparent verification reasoning, not objective truth determination.

## Validator Relationships

Validator relationships model how validator entities contribute verification decisions.

Concepts:
- a validator can evaluate one or more canonical records
- validator decisions are linked to specific provenance snapshots
- validator confidence and rationale are graph attributes, not standalone authority
- validator-to-validator relationships may exist as federation trust context, but do not imply automatic transitive trust

## Witness Relationships

Witness relationships model attestations bound to a canonical record and provenance context.

Concepts:
- a witness node references a canonical record identifier and related provenance signals
- a witness may attest to observation, process, or review outcome
- multiple witness nodes can attach to the same record without requiring consensus
- witness diversity (human + AI) is represented explicitly for downstream human-supervised validation

## Federation Topology Concepts

Federated verification is represented as topology, not as consensus finality.

Topology concepts:
- node clusters: independent verification infrastructure operators
- bridge edges: exchange paths for verification artifacts
- trust anchor references: known validator/witness identity roots
- sync boundaries: explicit scope of what is shared and what remains local

This topology prepares HC-TRUST-LAYER for federated verification while keeping local verification reproducible.

## Provenance Flow

Provenance flow links source context to verification outputs.

Example conceptual path:
1. canonical record established
2. provenance signals attached (origin, timestamp context, source references)
3. witness attestations added
4. validator checks executed
5. audit trail updated
6. trust graph view updated

Each edge in this flow should remain inspectable in verification infrastructure outputs.

## Audit Trail Graph Concepts

Audit trail in the trust graph is append-oriented and traceable.

Concepts:
- append-only event lineage for record status transitions
- supersession and revocation links as explicit edges
- review and re-verification events linked to prior states
- deterministic references to canonical record snapshots

## Replay Lineage Concepts

Replay lineage captures how a canonical record (or derivatives) reappears over time and distribution channels.

Concepts:
- replay edge: same or near-identical payload reintroduced in a later context
- lineage branch: derivative adaptation from prior canonical material
- conflict marker: contradictory provenance claims for similar content
- replay cluster: grouped events requiring human-supervised validation

## Trust Score Propagation Concepts

Trust score propagation is a model-level concept only.

Concepts:
- node-level trust signals contribute to edge-level confidence interpretation
- propagation should be bounded by evidence quality and validator/witness reliability history
- negative signals (revocation, inconsistency, disputed provenance) should dampen propagation
- no single witness or validator should create irreversible trust amplification

## Canonical Record Linkage

Canonical record linkage is the anchor for graph coherence.

Requirements:
- all trust graph entities reference a canonical record identifier
- provenance edges should include canonical payload hash context when available
- witness and validator artifacts should bind to canonical record boundaries
- cross-node federation exchange should preserve canonical linkage fidelity

## AI Witness vs Human Witness Distinction

HC-TRUST-LAYER distinguishes witness classes for interpretability.

AI witness:
- generated by automated model-assisted analysis
- useful for scale, pattern detection, and preliminary triage
- must remain reviewable and attributable

Human witness:
- produced via human-supervised validation or direct expert review
- provides contextual judgement and accountability signals
- required for sensitive escalation paths where policy requires human sign-off

Both classes can coexist in the trust graph with explicit witness type markers.

## Graph Integrity Considerations

Integrity of the trust graph depends on stable references and anti-tamper controls.

Core considerations:
- deterministic canonical record references
- immutable or append-traceable audit trail behavior
- explicit revocation/supersession edges
- integrity checks for orphaned or contradictory graph segments
- schema validation for graph artifact exchange

## Trust Graph Risks

Major risks include:
- false confidence from incomplete provenance
- over-weighting of a single validator or witness source
- ambiguity in replay lineage interpretation
- stale trust edges that no longer reflect current revocation state

## Graph Poisoning Considerations

Graph poisoning can occur when adversarial actors inject misleading or fabricated nodes/edges.

Mitigations to define in future protocol work:
- strict schema validation for graph inputs
- identity and key hygiene for witness/validator submissions
- anomaly detection for suspicious edge patterns
- quarantine workflow for contested graph artifacts

## Fake Validator Scenarios

Fake validator scenarios include impersonation or lookalike identities presenting fabricated validation outputs.

Mitigation concepts:
- validator identity binding to trust anchors
- signature verification and revocation checks
- federation allowlists and transparent validator registry metadata
- human-supervised validation for high-impact decisions

## Replay-Chain Manipulation Risks

Attackers may attempt to manipulate replay lineage by:
- reordering provenance events
- injecting forged intermediary records
- splitting one lineage into misleading parallel branches
- selectively omitting negative evidence links

Mitigation concepts:
- deterministic ordering rules for audit trail ingestion
- explicit parent/child lineage constraints
- contradiction detection during federated verification exchange
- mandatory linkage checks against canonical record references
