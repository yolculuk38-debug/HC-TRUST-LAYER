## Purpose

This document defines the advisory HC:// boundary between public verification data, runtime configuration data, signing material, and operational secrets. It is documentation only for HC-TRUST-LAYER operators and reviewers.

The boundary keeps public verification flows reviewable while preventing configuration or secret handling guidance from becoming runtime behavior. It does not implement secret managers, credentials, `.env` files (.env files), databases, infrastructure dependencies, validators, signing logic, workflows, schemas, federation behavior, or policy evaluator behavior.

## Configuration trust zones

HC:// operators should separate configuration into trust zones before review:

1. **Public verification zone**: deterministic public response fields, public-safe warnings, verification map references, provenance summaries, and audit trail pointers that are intended for transparent review.
2. **Runtime configuration zone**: deployment-specific settings that influence operator execution context but are not verification evidence by themselves.
3. **Signing custody zone**: private signing material and custody processes. This zone is outside public runtime output and outside documentation examples.
4. **Operational secret zone**: tokens, credentials, session material, infrastructure keys, and secret-manager handles. This zone is outside the trust kernel documentation surface unless described only as redacted operator responsibility.
5. **Human review zone**: human-supervised validation records, review notes, and escalation decisions that may reference restricted context without making it public.

Cross-zone movement should be minimized. If a public response needs to mention configuration, it should use public-safe status wording rather than raw values. If signing material or operational secrets appear in runtime output, logs, examples, or documentation, treat it as an exposure risk requiring operator review and remediation outside this PR.

## Secret exposure risks

Reviewers should treat the following as risks even when examples are non-functional:

- raw bearer tokens, API keys, session cookies, webhook secrets, cloud access keys, or private keys;
- private signing material, passphrases, seed phrases, or hardware-token unlock data;
- `.env` file contents or copied environment dumps;
- private infrastructure hostnames, internal paths, tenant identifiers, or credential scope names when not required for public review;
- logs that combine public verification data with private runtime configuration or secrets;
- screenshots, fixtures, or examples that use realistic credential formats.

Public-safe examples should use deterministic placeholders such as `<redacted-token>`, `<redacted-private-key>`, `<operator-config-name>`, and `<public-record-id>`. Placeholders must not be usable as credentials and must not resemble real tokens.

## Signing material handling guidance

Signing material is not public verification data. Public verification data may reference a signature result, key identifier, or provenance boundary only when that reference is already part of repository-defined public response semantics.

Operators remain responsible for:

- keeping private signing keys and unlock material outside the repository;
- separating signing custody from general runtime configuration;
- rotating or revoking material through reviewed operator procedures when exposure is suspected;
- documenting public impact without exposing private key material;
- preserving audit trail continuity and human-supervised validation during incident review.

This guidance does not implement key storage, key rotation automation, cryptographic enforcement, signing changes, or trust anchor changes.

## Operator responsibilities

Operators should maintain a local configuration inventory that distinguishes public verification data, runtime configuration, signing material, and operational secrets. The inventory should be reviewed before deployment, before sharing logs, and before publishing examples.

Minimum responsibilities:

- confirm public responses remain `advisory_only=true`, `public_safe=true`, and `truth_guarantee=false`;
- keep signing material and operational secrets out of docs, tests, runtime examples, logs, screenshots, and canonical record surfaces;
- redact secret-like data before sharing diagnostic material;
- keep runtime configuration separate from public verification evidence;
- use human-supervised validation for exposure triage and any operator-side remediation;
- avoid claiming production readiness, complete automation, or truth guarantees from configuration boundaries.

## Environment separation guidance

Use separate environments for local review, test/staging, and operator-managed deployment. Each environment should have distinct configuration names, distinct secret scopes, and an explicit public-safe output review step.

Recommended separation:

- **Local review**: uses placeholder configuration only; no real credentials or signing material.
- **Test or staging**: may use operator-managed non-production secrets outside the repository; public examples still use placeholders.
- **Operator-managed deployment**: uses external secret custody and infrastructure controls outside HC-TRUST-LAYER documentation; public verification output remains deterministic and redacted.

Do not add `.env` files (.env files), secret stores, Vault clients, Redis counters, JWT issuers, databases, or infrastructure dependencies as part of this documentation boundary.
