# v0.1.0 Release Checklist

Documentation-only v0.1.0 release checklist for HC-TRUST-LAYER. This checklist translates release governance into concrete pre-release, release, and post-release review steps for early-stage advisory HC:// verification infrastructure.

## 1. Purpose

This checklist helps contributors and reviewers prepare, validate, publish, and review the v0.1.0 release evidence package.

Repository evidence is authoritative. This checklist is advisory and documentation-only. v0.1.0 must remain framed as early-stage advisory HC:// verification infrastructure unless repository evidence supports a narrower or different claim.

## 2. Authority Boundary

This checklist does not create authority beyond repository evidence, repository-defined checks, reviewer decisions, and documented governance processes.

- This checklist does not create release approval authority.
- This checklist does not create merge authority.
- This checklist does not create security certification.
- This checklist does not create production readiness.
- This checklist does not create forensic certainty, truth finality, autonomous governance, live federation guarantees, complete dispute automation, final trust scoring semantics, final validator semantics, or cryptographic or policy guarantees beyond repository evidence.
- Human-supervised validation remains required for sensitive trust-kernel-impacting release content.

## 3. Release Scope

The intended v0.1.0 scope is:

- early-stage advisory HC:// verification infrastructure;
- repository evidence orientation;
- documentation, governance, validation, and release-control scaffolding;
- human-supervised validation boundaries;
- trust-kernel boundary documentation;
- validation architecture documentation;
- contributor and community workflow documentation.

## 4. Release Non-Goals

v0.1.0 does not claim or deliver:

- production readiness;
- security certification;
- forensic certainty;
- truth finality;
- autonomous governance;
- live federation guarantees;
- complete dispute automation;
- final trust scoring semantics;
- final validator semantics;
- cryptographic or policy guarantees beyond repository evidence.

## 5. Release Evidence Package

The v0.1.0 release evidence package should record the fields below before publication. Unknown or unavailable evidence should be marked explicitly rather than inferred.

| Evidence item | v0.1.0 expectation | Status |
| --- | --- | --- |
| Release version | `v0.1.0` | [ ] Recorded |
| Release candidate commit | Exact commit hash selected for release review | [ ] Recorded |
| Tag name | Expected release tag, normally `v0.1.0` | [ ] Recorded |
| Release notes path | `docs/v0.1.0-release-notes.md` | [ ] Recorded |
| Changelog section | `docs/releases.md` release-history entry or approved equivalent | [ ] Recorded |
| Changed files summary | Documentation, governance, validation, release-control, contributor, and community workflow changes summarized by path | [ ] Recorded |
| Checks run | Terminology, docs drift, canonical artifact, and whitespace checks recorded with outcomes | [ ] Recorded |
| Protected-path status | Confirmation that protected paths were unchanged, or reviewer-approved exceptions are documented | [ ] Recorded |
| Trust-kernel impact status | Documentation-only boundary impact status and required review routing | [ ] Recorded |
| Human-supervised validation status | Required, completed, not required, or pending, with reviewer evidence when sensitive content is involved | [ ] Recorded |
| Known limitations | v0.1.0 limitations and non-goals recorded or linked | [ ] Recorded |
| Approver / reviewer evidence | Reviewer names, approvals, review links, or other repository evidence | [ ] Recorded |

Minimum evidence-package checklist:

- [ ] Release version is recorded.
- [ ] Release candidate commit is recorded.
- [ ] Tag name is recorded.
- [ ] Release notes path is recorded.
- [ ] Changelog section is recorded.
- [ ] Changed files summary is recorded.
- [ ] Checks run are recorded.
- [ ] Protected-path status is recorded.
- [ ] Trust-kernel impact status is recorded.
- [ ] Human-supervised validation status is recorded.
- [ ] Known limitations are recorded.
- [ ] Approver / reviewer evidence is recorded.

## 6. Pre-Release Checklist

- [ ] Confirm the release candidate commit is selected and reviewable.
- [ ] Confirm the release version is `v0.1.0` unless reviewer-approved repository evidence supports a different version.
- [ ] Confirm the tag name is identified before publication.
- [ ] Confirm the release notes path is identified as `docs/v0.1.0-release-notes.md`.
- [ ] Confirm the changelog or release-history section is identified.
- [ ] Confirm the changed files summary distinguishes documentation-only changes from any protected-surface changes.
- [ ] Confirm release claims remain limited to repository evidence.
- [ ] Confirm no workflow, runtime, schema, validator, record, policy, federation, signing, trust-kernel index, `hc_context`, agent, source-code, test, `VERSION`, or `CHANGELOG.md` change is introduced unless separately approved.

## 7. Validation Checklist

- [ ] Run `python scripts/check_terminology.py`.
- [ ] Run `python scripts/check_docs_drift.py`.
- [ ] Run `python scripts/check_canonical_artifacts.py`.
- [ ] Run `git diff --check`.
- [ ] Record any unavailable check with the reason it could not run.
- [ ] Do not imply success for any check that did not run or did not pass.

## 8. Documentation Checklist

- [ ] Confirm release notes are current and consistent with the release evidence package.
- [ ] Confirm release governance references are followed.
- [ ] Confirm documentation uses HC-TRUST-LAYER, HC://, verification map, trust kernel, protocol graph, agent context, provenance, audit trail, canonical record, and human-supervised validation consistently when applicable.
- [ ] Confirm documentation avoids unsupported readiness, certification, certainty, finality, guarantee, or autonomous-governance claims.
- [ ] Confirm related documents are linked or identified for reviewer navigation.

## 9. Security Checklist

- [ ] Confirm the release does not claim security certification.
- [ ] Confirm security-sensitive claims are grounded in repository evidence.
- [ ] Confirm no security workflow change is included in the release checklist work unless separately approved.
- [ ] Confirm any security limitation is captured in release notes or known limitations.
- [ ] Confirm reviewer evidence is captured for any release content that may affect security interpretation.

## 10. Trust-Kernel Checklist

- [ ] Confirm whether release content affects trust-kernel boundary documentation.
- [ ] Confirm any trust-kernel-impacting content is routed for human-supervised validation.
- [ ] Confirm the release does not change trust-kernel indexes unless separately approved.
- [ ] Confirm canonical record, provenance, verification map, and protocol graph boundary language remains consistent with repository evidence.
- [ ] Confirm unresolved trust-kernel uncertainties are documented instead of inferred.

## 11. Protected-Surface Checklist

- [ ] Confirm no protected paths were modified unless explicitly approved.
- [ ] Confirm schema definitions were not changed.
- [ ] Confirm validators were not changed.
- [ ] Confirm signing and trust anchor semantics were not changed.
- [ ] Confirm federation behavior was not changed.
- [ ] Confirm policy evaluator behavior was not changed.
- [ ] Confirm canonical record surfaces were not changed.
- [ ] Confirm protected-path status is recorded in the release evidence package.

## 12. Human-Supervised Validation Checklist

- [ ] Identify whether the release content is documentation-only, trust-kernel-impacting, protected-surface-impacting, or policy-impacting.
- [ ] Record the expected human-supervised validation status.
- [ ] Capture reviewer or approver evidence before release publication when sensitive release content is involved.
- [ ] Confirm agent output remains advisory unless validated through repository-defined checks and reviewer oversight.
- [ ] Confirm release signoff does not replace repository review, required checks, protected-surface review, or human-supervised validation.

## 13. Changelog Checklist

- [ ] Confirm the changelog or release-history entry identifies `v0.1.0`.
- [ ] Confirm the entry summarizes documentation, governance, validation, release-control, contributor, and community workflow scope where applicable.
- [ ] Confirm the entry identifies known limitations or links to release notes that do.
- [ ] Confirm the entry does not claim production readiness, security certification, forensic certainty, truth finality, autonomous governance, live federation guarantees, complete dispute automation, final trust scoring semantics, final validator semantics, or unsupported cryptographic or policy guarantees.

## 14. Version and Tag Checklist

- [ ] Confirm the release version is recorded.
- [ ] Confirm the tag name is recorded.
- [ ] Confirm the tagged commit matches the reviewed release candidate commit.
- [ ] Confirm tag publication does not imply production readiness, security certification, forensic certainty, truth finality, autonomous governance, or guarantees beyond repository evidence.
- [ ] Confirm any version or tag correction path is documented if publication evidence is wrong.

## 15. Release Notes Checklist

- [ ] Confirm `docs/v0.1.0-release-notes.md` exists.
- [ ] Confirm release notes identify status, scope, limitations, and verification meaning.
- [ ] Confirm release notes frame v0.1.0 as early-stage advisory HC:// verification infrastructure unless repository evidence supports a narrower or different claim.
- [ ] Confirm release notes link or refer to relevant documentation for validation architecture, trust-kernel boundaries, contributor workflow, and community workflow when applicable.
- [ ] Confirm release notes do not create approval authority, merge authority, security certification, production readiness, or unsupported guarantees.

## 16. Publication Checklist

- [ ] Confirm release evidence package is complete or clearly marks unavailable evidence.
- [ ] Confirm required checks passed or are documented as unavailable with reasons.
- [ ] Confirm protected-path and trust-kernel impact statuses are recorded.
- [ ] Confirm human-supervised validation status is recorded.
- [ ] Confirm release notes and release-history references are ready for publication.
- [ ] Confirm reviewers have not objected to publication scope.

## 17. Post-Release Checklist

- [ ] Record the final tag and commit evidence.
- [ ] Record publication date and release notes location.
- [ ] Record any post-publication reviewer notes.
- [ ] Record any known follow-up actions.
- [ ] Confirm release artifacts remain consistent with repository evidence.
- [ ] Confirm any discovered inconsistency is routed through the correction process.

## 18. Rollback / Correction Checklist

- [ ] Identify the incorrect release artifact, tag, note, or claim.
- [ ] Preserve the audit trail for the correction.
- [ ] Document the reason for rollback, correction, supersession, or retirement.
- [ ] Confirm corrected language remains bounded by repository evidence.
- [ ] Confirm sensitive correction content receives human-supervised validation.
- [ ] Publish correction notes without implying erased history or unsupported authority.

## 19. Known Limitations

- v0.1.0 is early-stage advisory HC:// verification infrastructure.
- Repository evidence is authoritative, and this checklist does not expand repository guarantees.
- Release governance, version ownership, tag ownership, and release-control workflow may require additional formalization.
- Human-supervised validation remains required for sensitive trust-kernel-impacting release content.
- Final trust scoring semantics, final validator semantics, live federation guarantees, complete dispute automation, autonomous governance, truth finality, forensic certainty, security certification, and production readiness are non-goals for v0.1.0.
