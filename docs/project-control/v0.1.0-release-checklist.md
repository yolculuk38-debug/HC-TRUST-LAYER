# v0.1.0 Release Checklist

Documentation-only v0.1.0 release checklist for HC-TRUST-LAYER. This checklist translates release governance into concrete pre-release, release, and post-release review steps for early-stage advisory HC:// verification infrastructure.

## 1. Purpose

This checklist helps contributors and reviewers prepare, validate, publish, and review the v0.1.0 release evidence package.

Repository evidence is authoritative. This checklist is advisory and documentation-only. v0.1.0 must remain framed as early-stage advisory HC:// verification infrastructure unless repository evidence supports a narrower or different claim.

## 2. Authority Boundary

This checklist does not create authority beyond repository evidence, repository-defined checks, reviewer decisions, and documented governance processes.

- This checklist does not create release approval authority.
- This checklist does not create merge authority.
- This checklist does not create security certification.
- This checklist does not create production readiness.
- This checklist does not create forensic certainty, truth finality, autonomous governance, live federation guarantees, complete dispute automation, final trust scoring semantics, final validator semantics, or cryptographic or policy guarantees beyond repository evidence.
- Human-supervised validation remains required for sensitive trust-kernel-impacting release content.

## 3. Release Scope

The intended v0.1.0 scope is:

- early-stage advisory HC:// verification infrastructure;
- repository evidence orientation;
- documentation, governance, validation, and release-control scaffolding;
- human-supervised validation boundaries;
- trust-kernel boundary documentation;
- validation architecture documentation;
- contributor and community workflow documentation.

## 4. Release Non-Goals

v0.1.0 does not claim or deliver:

- production readiness;
- security certification;
- forensic certainty;
- truth finality;
- autonomous governance;
- live federation guarantees;
- complete dispute automation;
- final trust scoring semantics;
- final validator semantics;
- cryptographic or policy guarantees beyond repository evidence.

## 6. Pre-Release Checklist

- [ ] Confirm the release candidate commit is selected and reviewable.
- [ ] Confirm the release version is `v0.1.0` unless reviewer-approved repository evidence supports a different version.
- [ ] Confirm the tag name is identified before publication.
- [ ] Confirm the release notes path is identified as `docs/v0.1.0-release-notes.md`.
- [ ] Confirm the changelog or release-history section is identified.
- [ ] Confirm the changed files summary distinguishes documentation-only changes from any protected-surface changes.
- [ ] Confirm release claims remain limited to repository evidence.
- [ ] Confirm `docs/verify.html` is described only as a first-flow/demo static QR verification page and not as a verifier for arbitrary records.
- [ ] Confirm v0.1.0 does not claim a hosted general public verifier for arbitrary records.
- [ ] Confirm non-demo `/verify/{record_id}` QR routes are documented as advisory/navigation placeholders unless separately deployed and validated.
- [ ] Confirm existing QR artifacts are not treated as active v0.1.0 evidence unless decoded or regenerated after PR #592.
- [ ] Confirm public QR verification remains advisory-only and subject to human-supervised validation.
- [ ] Confirm no workflow, runtime, schema, validator, record, policy, federation, signing, trust-kernel index, `hc_context`, agent, source-code, test, `VERSION`, or `CHANGELOG.md` change is introduced unless separately approved.

## 18. Rollback / Correction Checklist

- [ ] Identify the incorrect release artifact, tag, note, or claim.
- [ ] Preserve the audit trail for the correction.
- [ ] Document the reason for rollback, correction, supersession, or retirement.
- [ ] Confirm corrected language remains bounded by repository evidence.
- [ ] Confirm sensitive correction content receives human-supervised validation.
- [ ] Publish correction notes without implying erased history or unsupported authority.

## 19. Known Limitations

- v0.1.0 is early-stage advisory HC:// verification infrastructure.
- Repository evidence is authoritative, and this checklist does not expand repository guarantees.
- Release governance, version ownership, tag ownership, and release-control workflow may require additional formalization.
- Human-supervised validation remains required for sensitive trust-kernel-impacting release content.
- `docs/verify.html` is a first-flow/demo static QR verification page only and does not verify arbitrary records.
- v0.1.0 does not provide a hosted general public verifier for arbitrary records.
- Non-demo `/verify/{record_id}` QR routes are advisory/navigation placeholders unless separately deployed and validated.
- Existing QR artifacts should not be treated as active v0.1.0 evidence unless decoded or regenerated after PR #592.
- Public QR verification remains advisory-only and subject to human-supervised validation.
- Final trust scoring semantics, final validator semantics, live federation guarantees, complete dispute automation, autonomous governance, truth finality, forensic certainty, security certification, and production readiness are non-goals for v0.1.0.
