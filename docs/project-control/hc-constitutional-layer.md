# HC Constitutional Layer v0.1

Advisory constitutional orientation for HC-TRUST-LAYER. This document is documentation only. It describes durable review expectations for HC:// repository governance without creating automation, enforcement, runtime behavior, merge authority, approval authority, security validation, production-readiness validation, truth-finality validation, or autonomous governance.

## 1. Purpose

The HC Constitutional Layer distinguishes between:

- foundational principles;
- constitutional principles;
- governance principles;
- operational principles;
- operational rules.

It exists to make clear which ideas are intended to be more stable, more durable, and more difficult to change over time. It helps reviewers reason about whether a proposal is changing a current task, an operating practice, a governance expectation, or a deeper HC:// constitutional boundary.

This layer is advisory only. It supports review, evidence collection, terminology consistency, provenance, audit trail continuity, and human-supervised validation expectations. It does not decide outcomes by itself.

## 2. Authority Boundary

- Repository evidence remains authoritative.
- Human-supervised validation remains required where repository rules require it.
- Repository-defined checks and reviewer oversight remain required where applicable.
- This Constitutional Layer is advisory only.
- This Constitutional Layer is documentation only.
- This Constitutional Layer is not approval authority.
- This Constitutional Layer is not merge authority.
- This Constitutional Layer is not governance enforcement.
- This Constitutional Layer is not security validation.
- This Constitutional Layer is not production-readiness validation.
- This Constitutional Layer is not truth-finality validation.
- This Constitutional Layer is not autonomous governance.

Nothing in this document grants reviewer power, maintainer power, policy authority, runtime authority, protected-path authority, trust-kernel authority, signing authority, federation authority, or workflow authority.

## 5. Constitutional Hierarchy

The HC Constitutional Layer uses the following durability hierarchy.

Most Stable:

1. Foundational Principles
2. Constitutional Principles
3. Governance Principles
4. Operational Principles
5. Operational Rules
6. Active Work
7. Current Priorities
8. Shift State

Most Changeable.

Higher layers should change less often, require stronger evidence, and receive more careful review. Lower layers may change more frequently as repository work progresses, but they must not quietly override higher-layer expectations or repository evidence.

## 6. Foundational Principles

Foundational Principles are the most durable advisory commitments in this layer. They describe baseline expectations for HC-TRUST-LAYER stewardship.

### 6.1 Repository Evidence First

Repository evidence comes first. Merged repository files, protected repository evidence, repository-defined checks, PR records, commits, review notes, and human review decisions are stronger than advisory summaries, chat memory, external claims, or unstated assumptions.

When repository evidence is incomplete, uncertain, or conflicting, the uncertainty should be surfaced rather than resolved through unsupported inference.

### 6.2 Human-Supervised Validation

HC-TRUST-LAYER depends on human-supervised validation for non-trivial trust-kernel-impacting changes and other sensitive boundaries identified by repository rules.

Agent output may assist drafting, review preparation, evidence gathering, and risk surfacing. Agent output does not replace reviewer oversight, repository-defined checks, or human-supervised validation.

### 6.3 Provenance and Audit Trail

HC:// reviewability depends on provenance and audit trail continuity. Changes should be attributable, reviewable, reversible where possible, and supported by repository evidence.

Decision records, PR history, commit history, check outputs, and reviewer notes should preserve enough context for later review without relying on hidden authority or inaccessible memory.

### 6.4 Terminology Consistency

HC-TRUST-LAYER terminology should remain consistent across repository-facing content. Terms including HC://, verification map, trust kernel, protocol graph, agent context, provenance, audit trail, canonical record, and human-supervised validation should be preserved unless a reviewed repository change updates their meaning.

Terminology consistency supports transparent review boundaries and reduces constitutional trust risk caused by ambiguous language.

### 6.5 Advisory AI Principle

AI assistance in HC-TRUST-LAYER is advisory. It may help summarize repository evidence, draft documentation, identify possible risks, and prepare review artifacts.

AI assistance does not create approval authority, merge authority, governance enforcement, security validation, production-readiness validation, truth-finality validation, or autonomous governance.

## 7. Constitutional Principles

Constitutional Principles are durable advisory rules for interpreting governance legitimacy, authority boundaries, evidence, and change review.

### 7.1 Auditability Principle

Governance guidance, decisions, exceptions, and changes should be auditable through repository evidence. Authority claims should be traceable to reviewed repository records rather than hidden context or unverifiable assertions.

Auditability is an expectation for reviewability. It is not enforcement by itself.

### 7.2 Transparency Principle

Constitutional and governance decisions should be understandable to future reviewers. The reason for a decision, the evidence used, the affected boundary, and the expected follow-up should be visible in repository records where practical.

Opaque decision paths, undocumented exceptions, and non-reviewable authority claims create constitutional trust risk.

### 7.3 Accountability Principle

Responsibility and authority should remain linked to traceable repository evidence. Mistakes, limitations, and reversals should be documented when relevant so later reviewers can understand what changed and why.

Accountability is not a guarantee of correctness. It is an expectation that decisions remain attributable and reviewable.

### 7.4 Evidence First Principle

Constitutional interpretation should start from repository evidence before advisory summaries or external assumptions. Claims about implementation status, protected paths, trust-kernel boundaries, policy behavior, schema behavior, validator behavior, signing behavior, federation behavior, or runtime behavior require repository support.

Unsupported claims should be marked as advisory, proposed, pending, or unknown.

### 7.5 Documentation-Only Principle

This Constitutional Layer is documentation only. It does not modify runtime behavior, workflow behavior, schemas, validators, records, policy files, federation logic, signing logic, trust-kernel indexes, canonical record surfaces, or protected-path behavior.

Documentation may describe expectations, but it does not execute them.

### 7.6 Advisory-Only Principle

This Constitutional Layer is advisory only. It may inform discussion, review preparation, and decision records, but it does not approve, merge, enforce, validate, or finalize governance outcomes.

Where repository evidence or human review conflicts with this advisory layer, stronger repository evidence and reviewed decisions control.

### 7.7 No Hidden Authority Principle

No individual, founder, maintainer, reviewer, operator, governance actor, contributor, organization, institution, company, future automation, or external authority is exempt from:

- repository evidence;
- review requirements;
- protected-path boundaries;
- human-supervised validation expectations.

Authority that cannot be reviewed, traced, attributed, or evidenced creates constitutional trust risk.

### 7.8 Equality Principle

Different roles may have different responsibilities, permissions, authority scopes, and review obligations.

Governance requirements, evidence requirements, audit expectations, protected-path boundaries, and validation expectations apply consistently regardless of role.

The Equality Principle does not erase role differences. It requires that role differences remain reviewable, evidenced, and bounded by repository-defined expectations.

## 8. Governance Principles

Governance Principles translate constitutional expectations into review-oriented governance guidance.

- Governance changes should be scoped, reversible where possible, and auditable.
- Governance proposals should identify affected repository surfaces and expected decision-path differences.
- Governance documentation should preserve HC:// terminology and avoid unsupported authority claims.
- Protected-path and trust-kernel-adjacent proposals should surface human-supervised validation needs before merge.
- Governance records should preserve provenance and audit trail continuity.
- Advisory AI may support governance work but must not become governance enforcement.
- Governance documents may describe review expectations but must not claim approval authority or merge authority.

## 9. Operational Principles

Operational Principles guide day-to-day repository work without overriding constitutional or governance layers.

- Prefer small, scoped, reviewable changes.
- Prefer documentation-first clarification before behavior changes.
- Preserve existing guardrails, checks, and protected-path boundaries.
- Report changed files, checks run, commit hashes, PR numbers, and known limitations.
- Separate follow-up work from the current change when possible.
- Treat active work, current priorities, and shift state as changeable coordination surfaces.
- Avoid production-readiness, truth guarantee, forensic certainty, autonomous governance, or misleading authority claims unless supported by repository evidence and validation.

Operational rules may be updated more frequently than constitutional material, but they should not contradict repository evidence or higher-layer constitutional expectations.

## 10. Constitutional Boundaries

Constitutional boundaries are areas where changes may affect durable repository trust expectations even when the file changed is documentation-only.

Constitutional risk should be considered when a proposal affects or reinterprets:

- trust-kernel boundaries;
- protected-path definitions;
- canonical record boundaries;
- verification map meaning;
- protocol graph meaning;
- provenance continuity;
- audit trail continuity;
- policy interpretation;
- workflow governance;
- schema governance;
- validator governance;
- signing governance;
- federation governance;
- human-supervised validation expectations;
- advisory AI boundary claims.

This document does not change any of those boundaries. It only identifies them as review-sensitive concepts.

## 11. Amendment Expectations

Amendments to this Constitutional Layer should be classified before review.

### 11.1 Amendment Classes

- **Editorial:** Fixes spelling, formatting, links, or wording without changing meaning.
- **Clarifying:** Improves explanation or resolves ambiguity without changing expected governance interpretation.
- **Governance-Adjacent:** Affects governance guidance, review routing, protected-surface descriptions, or operational governance expectations.
- **Constitutional:** Changes foundational or constitutional principles, source-of-truth order, authority boundaries, equality expectations, or decision outcomes.
- **Trust-Kernel-Sensitive:** Affects or reinterprets trust-kernel boundaries, canonical record boundaries, schema governance, validator governance, signing governance, federation governance, policy interpretation, provenance continuity, or human-supervised validation expectations.

### 11.2 Amendment Expectations by Class

- Editorial amendments may use lightweight review when repository rules allow.
- Clarifying amendments should cite the ambiguity being resolved.
- Governance-Adjacent amendments should identify affected governance documents and expected review impacts.
- Constitutional amendments should provide strong repository evidence, explicit rationale, and reviewer visibility.
- Trust-Kernel-Sensitive amendments require explicit human-supervised validation planning and should not be treated as routine documentation edits.

## 12. Constitutional Review Model

Constitutional review should ask:

1. What layer is being changed: foundational, constitutional, governance, operational, or active-work coordination?
2. What repository evidence supports the change?
3. What protected paths, trust-kernel boundaries, or canonical record boundaries are affected or adjacent?
4. Does the change alter authority, approval, merge, enforcement, validation, or truth-finality expectations?
5. Does the change preserve provenance and audit trail continuity?
6. Does the change require human-supervised validation?
7. What decision outcome should be recorded?

### 12.1 Constitutional Decision Outcomes

- **Draft:** Proposed text or concept not yet ready for decision.
- **Under Review:** Actively being evaluated by reviewers or maintainers.
- **Accepted:** Approved through repository review within the documented scope.
- **Accepted with Limits:** Accepted only for a bounded scope, timeframe, interpretation, or follow-up condition.
- **Deferred:** Not decided now; may return when evidence, priority, or review capacity changes.
- **Rejected:** Not accepted based on current evidence or review.
- **Superseded:** Replaced by later repository evidence or reviewed guidance.
- **Retired:** No longer active guidance, while historical provenance remains preserved.

## 13. Evidence Requirements

Constitutional or governance-adjacent proposals should identify relevant evidence, such as:

- changed files;
- linked repository documentation;
- protected-path assessment;
- trust-kernel impact assessment;
- check outputs;
- PR records;
- commit hashes;
- review notes;
- decision records;
- known limitations;
- follow-up tasks.

Evidence should be sufficient for a future reviewer to understand what changed, why it changed, what authority was claimed, what authority was not claimed, and what validation remains pending if any.

## 14. Human-Supervised Validation Expectations

Human-supervised validation expectations should be surfaced when a change affects or reinterprets:

- trust-kernel boundaries;
- protected-path definitions;
- policy interpretation;
- workflow governance;
- schema governance;
- validator governance;
- signing governance;
- federation governance;
- provenance continuity.

For each affected area, review records should identify:

- the affected surface;
- the expected decision-path difference;
- applicable repository-defined checks;
- whether protected-path or trust-kernel review is needed;
- whether validation is complete, limited, pending, deferred, or not applicable;
- any residual risks or guardrails.

This Constitutional Layer does not perform human-supervised validation. It only documents expectations for surfacing that validation need.

## 15. Constitutional Decision Model

A constitutional decision should be recorded with enough context to preserve audit trail continuity.

Recommended decision fields are:

- decision title;
- decision outcome;
- amendment class;
- affected hierarchy layer;
- affected repository surfaces;
- evidence summary;
- authority boundary statement;
- validation expectation;
- review notes;
- follow-up or retirement condition.

The model is advisory and documentation-only. It does not create a required workflow, automated gate, or approval mechanism.

## 16. Source-of-Truth Model

When constitutional guidance, governance guidance, project-control material, advisory context, or external summaries conflict, use this source-of-truth order:

1. Merged repository files and protected repository evidence.
2. Repository-defined checks and validation outputs.
3. PR records, commits, review notes, and human review decisions.
4. Protected-path and trust-kernel boundaries documented in repository evidence.
5. Constitutional Layer.
6. Governance Change Protocol.
7. Project-control documentation.
8. Advisory context and external summaries.

This order does not let the Constitutional Layer override stronger repository evidence. It places this layer above ordinary project-control guidance only as advisory constitutional interpretation.

## 17. Constitutional Memory Model

Constitutional memory is the repository-preserved record of durable principles, authority boundaries, decision outcomes, and amendment history.

Constitutional memory should be:

- repository-native;
- reviewable;
- attributable;
- versioned through commits and PRs;
- connected to decision records when relevant;
- independent of hidden chat memory or external summaries;
- clear about superseded and retired guidance.

Advisory agent context may help summarize constitutional memory, but it must not become the canonical record of that memory.

## 18. Relationship to HC Control Center

The HC Control Center is an operator orientation entry point. This Constitutional Layer gives deeper advisory context for durable principles and authority boundaries.

The HC Control Center should remain concise and operational. It should point to repository evidence and avoid replacing this Constitutional Layer, repository-defined checks, reviewer oversight, or human-supervised validation.

## 19. Relationship to Active Work Registry

The Active Work Registry is a changeable coordination surface for current work. It belongs near the most changeable end of the constitutional hierarchy.

Active work entries should not override foundational principles, constitutional principles, governance principles, repository evidence, protected-path boundaries, or human-supervised validation expectations.

## 20. Relationship to Project State

Project State summarizes the current repository phase, active focus, parked work, and status context. It is more changeable than the Constitutional Layer.

Project State may cite constitutional expectations, but it should not redefine authority boundaries or source-of-truth order without a reviewed constitutional or governance change.

## 21. Relationship to Task Ledger

The Task Ledger preserves task history, completed references, do-not-repeat notes, and review-oriented continuity. It supports provenance and audit trail continuity.

The Task Ledger may provide evidence for constitutional review, but it does not replace merged repository files, repository-defined checks, PR records, or human review decisions.

## 22. Relationship to Next Actions

Next Actions identifies safe, priority-ordered work. It is operational and changeable.

Next Actions should remain consistent with constitutional and governance boundaries, especially when tasks are report-only, docs-only, protected-path-adjacent, or trust-kernel-sensitive.

## 23. Relationship to Governance Change Protocol

The Governance Change Protocol describes how governance-oriented documentation may be proposed, reviewed, decided, superseded, or retired.

This Constitutional Layer provides the higher-level advisory hierarchy and principles used to classify whether a governance change is routine, governance-adjacent, constitutional, or trust-kernel-sensitive.

## 24. Relationship to Decision Registry

The Decision Registry is the preferred project-control surface for recording durable decisions and their outcomes when a separate record is useful.

Constitutional decisions should be reflected in the Decision Registry when they materially affect authority boundaries, amendment classes, source-of-truth interpretation, protected-path expectations, or human-supervised validation expectations.

## 25. Relationship to HC_BOOTSTRAP

HC_BOOTSTRAP is a first-read operational handoff surface. It helps operators begin from repository-native context.

This Constitutional Layer is deeper advisory context. HC_BOOTSTRAP should not be treated as a replacement for constitutional principles, repository evidence, checks, or reviewer decisions.

## 26. Relationship to AGENTS.md

AGENTS.md provides repository-wide contributor and agent instructions, terminology baseline, safe task boundaries, and required checks. It is direct repository guidance for agents and contributors.

This Constitutional Layer does not override AGENTS.md. When AGENTS.md and this layer both apply, contributors should follow AGENTS.md operational instructions while using this layer as advisory constitutional context.

## 27. Relationship to HC Guide Bot

The future HC Guide Bot should remain advisory. It may orient users to repository evidence, summarize project-control documentation, surface likely protected-path issues, and warn about human-supervised validation expectations.

The HC Guide Bot must not claim approval authority, merge authority, governance enforcement, security validation, production-readiness validation, truth-finality validation, or autonomous governance.

## 28. Relationship to future HC Control Bot

A future HC Control Bot may assist with operator flows, check diagnosis, PR scope inspection, verification package inspection, rerun guidance, and audit or recovery summaries.

Any such bot must remain bounded, explainable, reversible where applicable, and human-reviewable. It must not auto-merge or enforce governance, especially for security-sensitive, policy-sensitive, validator, signing, federation, schema, canonical record, protected-path, or trust-kernel-sensitive changes.

## 29. Relationship to future HC Audit Layer

A future HC Audit Layer may help organize evidence, audit trails, decision histories, and provenance continuity.

The future HC Audit Layer should support reviewability without claiming forensic certainty, truth-finality validation, autonomous governance, or security validation unless such claims are implemented, validated, and documented in repository evidence.
