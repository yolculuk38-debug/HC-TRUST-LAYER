# Source Top-Level Module Boundary Audit

## Purpose

This document completes the architect-audit backlog item for the current top-level `src/` module surface. It creates an evidence-first map before any refactor, move, delete, archive, namespace split, cleanup, or retirement work.

A factory labels machines before moving them. HC should classify modules before moving or retiring code.

## HC boundary statement

- `advisory_only=true`
- `public_safe=true`
- `truth_guarantee=false`
- `human_review_required=true`
- `approval_authority=false`
- `merge_authority=false`
- `label_reviewer_mutation=false`
- `issue_comment_automation=false`
- no automatic comments
- no automatic labels
- no reviewer requests
- no approvals
- no merges
- no runtime behavior change
- no workflow behavior change
- no schema/validator/record/generated/canonical behavior change

This audit is advisory-only. It does not move, delete, rename, archive, split, or refactor code. It does not change runtime behavior. It does not change public API or CLI behavior. It does not change schema, validator, record, generated, canonical, workflow, policy, federation, signing, hash, or QR behavior. CI/checks are evidence, not trust authority. Human maintainers make the final decision.

## Scope and non-effects

In scope:

- visible top-level files and directories directly under `src/`;
- `pyproject.toml` as package-discovery evidence only;
- `docs/repo-map.md` and existing project-control boundary reviews as documentation evidence;
- `tests/**` only as a coverage signal by visible module/name reference.

Out of scope:

- source edits;
- test edits;
- workflow edits;
- schema, validator, record, generated, canonical, policy, federation, signing, hash, or QR edits;
- root governance edits;
- import rewrites;
- namespace migration;
- runtime, public API, or CLI behavior changes.

## Method

The audit inspected the visible top-level `src/` inventory, then classified each item by current path/name evidence and by the repository's existing boundary documents. The evidence basis is intentionally conservative: path names and existing project-control documents can support review triage, but they do not prove hidden maintainer intent or final architecture.

Coverage signals were checked by searching `tests/**` for matching module or package names. A visible test signal means only that related test text was found; it does not prove complete behavioral coverage. No test, check, or CI result grants trust authority.

## Initial classification

The classification vocabulary is limited to the requested categories:

- core package surface;
- support/helper surface;
- governance/ops support;
- demo/example-adjacent;
- experimental/research;
- relocation candidate;
- retirement candidate;
- needs more review.

This audit found many current practical repo-state modules that appear active or at least referenced. No top-level item is marked as an immediate retirement candidate because this PR did not inspect semantic reachability deeply enough to support that recommendation. Demo/example-adjacent and experimental/research items may become relocation candidates later, but only after maintainer review and tests/docs-only follow-up where possible.
