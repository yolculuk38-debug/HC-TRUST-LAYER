> - **status:** DRAFT

## Purpose

This document defines how HC:// transitions from the Foundation/Governance baseline into an Operational Core that is auditable, challengeable, and aligned with human-supervised validation.

It preserves advisory-only verification and does not claim production readiness for unfinished systems.

Repository evidence supports these completed architecture layers as transition prerequisites:

Operational Core development should prioritize the following goals:

- validator orchestration
- federation synchronization

- verification lifecycle management
- public verification APIs

## Operational Core Components

Operational Core implementation should be organized into bounded, reviewable components:

- validator coordinator
- federation sync engine
- audit continuity engine
- trust scoring engine
- challenge/dispute router
- evidence preservation layer
- public verification gateway
- reviewer escalation pipeline

## Architectural Priorities

HC-TRUST-LAYER operational evolution should preserve this priority order:

1. reliability before scale
2. inspectability before automation
3. traceability before speed
4. accountability before convenience
5. federation before centralization

## Transition Phases

### 1) Foundation Stable Candidate

- Foundation/Governance meaning remains consistent and inspectable.
- Advisory-only verification and human-supervised validation remain mandatory.
- Trust-kernel boundaries remain explicit and documentation-first.

### 2) Controlled Operational Prototype

- Introduce bounded validator orchestration and verification routing.
- Keep disagreement visible through explicit conflict-state handling.
- Preserve reviewer authority over consequential interpretation.

### 3) Federation Expansion

- Expand federation synchronization and cross-review participation.
- Surface divergence, uncertainty, and escalation paths as first-class signals.
- Preserve attributable provenance across participants.

### 4) Public Verification Scaling

- Expand public verification APIs and reviewer-facing support tooling.
- Improve mobile-readable verification flow and caution-state clarity.
- Preserve transparency and challengeability before automation depth.

### 5) Distributed Trust Infrastructure

- Coordinate validator ecosystem interoperability at broader federation scale.
- Keep audit continuity and evidence preservation mandatory.
- Preserve governance oversight and human-supervised validation as non-bypassable.

## Operational Risks

Operational growth must continuously track these risks:

- orchestration complexity
- federation divergence
- false trust amplification
- reviewer overload
- AI overreach
- governance fragmentation
- hidden operational shortcuts

## Governance and Control Constraints

Operational growth must remain controlled through phased rollout, reversible increments, and explicit trust-impact review.

Governance must evolve with infrastructure so policy interpretation, escalation routing, and oversight accountability remain synchronized with new operational surfaces.

No hidden authority layer should emerge; consequential interpretation must remain challengeable, attributable, and human-supervised.

Operational systems must remain auditable so provenance continuity, dispute chronology, and reviewer accountability are preserved across the lifecycle.

## Progression Model

HC:// progression should remain staged in this order:

1. repository
2. protocol
3. governance architecture
4. operational core
5. validator ecosystem
6. federated trust network
