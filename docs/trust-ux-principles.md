# HC-TRUST-LAYER HC:// Trust UX Principles

## Purpose

These principles define a simplicity-first trust UX baseline for HC-TRUST-LAYER and HC:// interaction-layer planning.

## Simplicity-First Trust UX

Trust UX should reveal verification signals in the smallest understandable form before advanced technical depth.

## Non-Technical Readability

Core verification meaning should be understandable by non-technical users without requiring protocol specialization.

## Fast Comprehension Goals

Critical trust signals should be recognizable within seconds:

- what was verified
- what is uncertain
- what requires human-supervised validation

## Low Terminology Overload

Use canonical terms only when needed and avoid dense jargon clusters in primary views.

Secondary detail layers may include technical context, but core views should remain plain-language first.

## Trust-Oriented Language

Use language that communicates verification boundaries accurately:

- signal visibility over certainty inflation
- evidence-linked interpretation over unsupported conclusion
- explicit boundaries for automated outputs

## Warning Clarity

Warnings should be visually and textually clear, especially for replay, provenance gaps, or dispute states.

Each warning should include what it means, why it matters, and what escalation action is recommended.

## Confidence Communication

Confidence framing should be careful and bounded.

Interfaces should not imply production certainty or autonomous governance finality beyond documented evidence.

## Mobile-First Interaction

Trust UX should prioritize mobile verification contexts with:

- concise summaries
- touch-friendly controls
- progressive disclosure
- low-friction transitions from summary to evidence

## Accessible Verification Concepts

Verification concepts should remain accessible through:

- short labels
- glossary-linked hints
- consistent icon and status semantics
- fallback non-graph views where visual complexity is high
